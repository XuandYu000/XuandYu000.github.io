This is actually allowed in Java! It's something called method overloading. When you call WordUtils.longest, Java knows which one to run according to what kind of parameter you supply it. If you supply it with an AList, it will call the AList method. Same with an SLList.

这在java中是允许的, 被称为函数重载. 当你调用WordUtils.longest, java会根据参数类型来得知你调用的是哪一个. 如果参数类型是AList, 会调用AList函数. SLList同样

It's nice that Java is smart enough to know how to deal with two of the same methods for different types, but overloading has several downsides:

java知道怎么处理有不同类型参数的相同方法这个特性是极好的, 但是重载也有些副作用:

  • It's super repetitive and ugly, because you now have two virtually identical blocks of code. 这是非常繁琐和丑陋的, 因为你现在有2个看起来一样的代码块

  • It's more code to maintain, meaning if you want to make a small change to the longest method such as correcting a bug, you need to change it in the method for each type of list. 有更多的代码要去维护, 意味着如果你想要在longest方法做出改变比如修正个bug之类的, 你需要改变每个类型的方法

  • If we want to make more list types, we would have to copy the method for every new list class. 如果我们想制作更多列表类型，则必须为每个新列表类复制该方法。

Hypernyms(上位词), Hyponyms(下位词), and Interface Inheritance(接口继承)

In the English language and life in general, there exist logical hierarchies(层次结构) to words and objects.

Dog is what is called a hypernym of poodle, malamute, husky, etc. In the reverse direction, poodle, malamute, and husky, are hyponyms of dog.

狗是poodle(贵宾犬), malamute(阿拉斯加雪橇犬), 哈士奇这类词的上位词. 同样的逻辑, 这类词是狗的下位词

These words form a hierarchy of "is-a" relationships:

  • a poodle "is-a" dog
  • a dog "is-a" canine(犬科动物)
  • a canine "is-a" carnivore(肉食动物)
  • a carnivore "is-an" animal

The same hierarchy goes for SLLists and ALists! SLList and AList are both hyponyms of a more general list.

We will formalize this relationship in Java: if a SLList is a hyponym of List61B, then the SLList class is a subclass of the List61B class and the List61B class is a superclass of the SLList class.

Figure 4.1.1

In Java, in order to express this hierarchy, we need to do two things: 在java中为了表示这种层次关系, 我们需要做两件事

  • Step 1: Define a type for the general list hypernym -- we will choose the name List61B. 步骤1：定义通用列表上位词的类型 - 我们将选择名称list61b。

  • Step 2: Specify that SLList and AList are hyponyms of that type. 指定SLList和AList是这种类型二点下位词

The new List61B is what Java calls an interface. It is essentially a contract that specifies what a list must be able to do, but it doesn't provide any implementation for those behaviors. Can you think of why? 新的List61B在java中被称为接口. 这是一种很重要的压缩指明了一个list必须能够做什么, 但是它却不会提供这些行为的实现方法. 你能想一想这是为什么吗?

Here is our List61B interface. At this point, we have satisfied the first step in establishing the relationship hierarchy: creating a hypernym. 这是我们的List61B接口. 在这里, 我们满足了表示层次结构的第一点: 建立上位词

Now, to complete step 2, we need to specify that AList and SLList are hyponyms of the List61B class. In Java, we define this relationship in the class definition. 现在来实现step 2, 我们要指明AList和SLList是List61B二点下位词. 在java中我们在类定义中表明这种关系.

We will add to

public class AList<Item> {...}

a relationship-defining word: implements.

public class AList<Item> implements List61B<Item>{...}

implements List61B<Item> is essentially(本质上) a promise. AList is saying "I promise I will have and define all the attributes and behaviors specified in the List61B interface" implements List61B<Item>本质上是一个承诺. AList说"我保证会实现List61B中的所有属性和方法"

Now we can edit our longest method in WordUtils to take in a List61B. Because AList and SLList share an "is-a" relationship.

Overriding

We promised we would implement the methods specified in List61B in the AList and SLList classes, so let's go ahead and do that. 我们保证在AList和SLList实现List61B中指明的方法.

When implementing the required functions in the subclass, it's useful (and actually required in 61B) to include the @Override tag right on top of the method signature. Here, we have done that for just one method. 当实现子类需要的函数时, 在方法签名上包括@override标签是非常有用的(实际上在61B也是需要的).这里我们已经在一个方法上完成它

It is good to note that even if you don’t include this tag, you are still overriding the method. So technically, you don't have to include it. However, including the tag acts as a safeguard for you as the programmer by alerting the compiler that you intend to override this method. Why would this be helpful you ask? Well, it's kind of like having a proofreader! The compiler will tell you if something goes wrong in the process. 这样做是有好处的即使你不包含这个标签你也依然能够重载这个方法. 所以从技术上讲, 你不必包含它. 然而这个标签实际上扮演着保险的角色来告诉编译器你会重载这个方法. 你会问这为什么是有用的? 实际上这是一个校验器! 如果在这个过程中出错编译器会告诉你

Say you want to override the addLast method. What if you make a typo(错别字) and accidentally write addLsat? If you don't include the @Override tag, then you might not catch the mistake, which could make debugging a more difficult and painful process. Whereas if you include @Override, the compiler will stop and prompt you to fix your mistakes before your program even runs. 加入你想重载addLast这个方法. 如果你拼错实际上偶然写成了addLsat了呢? 如果你没有包括@Overrde这个标签, 你可能不会发现这个错误,但这会使debuging更困难且更具危害性. 然而如果你包括了这个标签, 编译器甚至会在你的运行之前停止且会提示你解决错误.

Interface Inheritance

Interface Inheritance refers to a relationship in which a subclass inherits all the methods/behaviors of the superclass. As in the List61B class we defined in the Hyponyms and Hypernyms section, the interface includes all the method signatures, but not implementations. It's up to the subclass to actually provide those implementations. 接口继承是指子类继承父类的所有方法/行为. 就像我们在List61B中定义的下位词和上位词部分, 接口包括所有的方法和签名, 但是却没有实现. 这取决于子类来实际提供这些实现.

This inheritance is also multi-generational. This means if we have a long lineage of superclass/subclass relationships like in Figure 4.1.1, AList not only inherits the methods from List61B but also every other class above it all the way to the highest superclass AKA AList inherits from Collection. 这种继承也是多代的. 这意味着如果我们有着一条像Figure 4.1.1很长的关系链, AList不仅仅会继承来自List61B中的方法也会继承其上位关系的方法,直至到最上层的父类

You may be wondering, how does Java know which print() to call? Good question. Java is able to do this due to something called dynamic method selection(动态方法选择).

We know that variables in java have a type.

List61B<String> lst = new SLList<String>();

In the above declaration and instantiation, lst is of type "List61B". This is called the "static type"

However, the objects themselves have types as well. the object that lst points to is of type SLList. Although this object is intrinsically(本质上) an SLList (since it was declared as such), it is also a List61B, because of the “is-a” relationship we explored earlier. But, because the object itself was instantiated using the SLList constructor, We call this its "dynamic type".

Aside: the name “dynamic type” is actually quite semantic in its origin! Should lst be reassigned to point to an object of another type, say a AList object, lst’s dynamic type would now be AList and not SLList! It’s dynamic because it changes based on the type of the object it’s currently referring to.

When Java runs a method that is overriden, it searches for the appropriate method signature in it's dynamic type and runs it.

IMPORTANT: This does not work for overloaded methods!

Interface Inheritance vs Implementation Inheritance

How do we differentiate between "interface inheritance" and "implementation inheritance"? Well, you can use this simple distinction: 我们怎么区别"接口继承"和"实现继承"呢？ 你可以通过这些简单的差别区分

  • Interface inheritance (what): Simply tells what the subclasses should be able to do. 接口继承(什么): 告诉你子类应该能做什么.
    • EX) all lists should be able to print themselves, how they do it is up to them.
    • 比如 所有的lists应该能够打印自己, 怎么做取决于它们自己
  • Implementation inheritance (how): Tells the subclasses how they should behave.实现继承(怎么): 告诉子类他们应该做什么
    • EX) Lists should print themselves exactly this way: by getting each element in order and then printing them.
    • 比如lists应该这样打印自己: 将所有元素排序之后打印他们

When you are creating these hierarchies, remember that the relationship between a subclass and a superclass should be an "is-a" relationship. AKA Cat should only implement Animal Cat is an Animal. You should not be defining them using a "has-a" relationship. Cat has-a Claw, but Cat definitely should not be implementing Claw.

Finally, Implementation inheritance may sound nice and all but there are some drawbacks: 最后, 实现继承可能听起来很好且全面但是这里有一些缺点:

  • We are fallible(容易犯错) humans, and we can't keep track of everything, so it's possible that you overrode a method but forgot you did. 我们是易犯错的人, 而且我们不能追踪所有事, 所以可能你覆盖了一个方法但是却忘记你做了
  • It may be hard to resolve conflicts in case two interfaces give conflicting default methods. 当遇到两个接口给出冲突的默认方法时冲突很难解决
  • It encourages overly complex code 这会导致代码复杂化
